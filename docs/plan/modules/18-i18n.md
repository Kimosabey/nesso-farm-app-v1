# Module · Internationalization (12 languages)

## Purpose
First-class support for 12 languages across mobile, web, and QR portal. English is the source of truth; all other locales are derived.

## Library
- **i18next** + **react-i18next** (web, mobile)
- Web: `next-intl` or i18next's `next-i18next` (decide in Phase 0); App Router prefers locale-prefix paths
- Portal: locale prefix `/en`, `/hi`, ...
- Dashboard: locale stored in user preferences (server) — applied client-side via i18next; URL not locale-prefixed for the gated dashboard

## Key strategy
- Dot-namespaced (`auth.otp.title`, `farmer.fields.mobileNumber`)
- ICU MessageFormat for interpolation and plurals: `"farmer.count": "{count, plural, one {# farmer} other {# farmers}}"`
- Never concatenate fragments
- Long strings broken into `*.line1`, `*.line2` only when necessary; prefer single keys with interpolation

## Tooling

### `pnpm i18n:scan`
- Scans `apps/*/src/**` for `t('...')` calls
- Flags keys present in code but missing in `en`
- Flags keys orphaned in `en` but not used anywhere

### `pnpm i18n:check`
Runs in CI:
- Verifies every non-en locale has the same key set as `en` (missing keys allowed, but reported per-locale)
- Verifies no leftover corruption markers (escaped JSON, mojibake)
- Verifies ICU plural categories valid per language

### `pnpm i18n:translate`
- Optional: pipes missing keys to a translation provider (DeepL / Azure Translator) with per-language glossaries
- Always creates a PR for human review; never auto-merges

## Fallback chain
- Resource missing in `kn` → falls back to `en`
- Never displays the raw key (`farmer.fields.mobileNumber`); shows the English fallback instead

## Layout considerations
- Text expansion budget: 1.4× English line lengths (validated in Storybook with `dummyLang`)
- Numbers, currencies, dates via `Intl.*` (`Intl.NumberFormat`, `Intl.DateTimeFormat`)
- Numerals can be locale-native (`१२३४५` in Hindi) — togglable in user preferences; default Western

## Language switching
- Web: `next-intl` rewrites the path; full page reload (RSC) for new locale to avoid hydration mismatch
- Mobile: `i18n.changeLanguage(code)`; subscribes to `expo-localization.getLocales()` for OS toggles
- Persisted to user prefs (server) so the choice is consistent across devices

## RTL
- No current languages are RTL, but the system uses logical CSS properties (`padding-inline-start`) so adding Arabic later is a content task, not a layout rewrite
- React Native: `I18nManager.allowRTL(true)` ready to flip

## A11y
- `<html lang>` set per locale (web/portal)
- `accessibilityLanguage` set on mobile screens via `react-native-localize`

## Edge cases
- Mixed-script content (Kannada farm name in an English UI) renders fine — fonts include CJK + Devanagari + Indic ranges
- Long IDs and codes (Latin) stay in monospace regardless of locale

## Acceptance criteria
- AC1: Switching language on mobile changes UI within 200 ms.
- AC2: Web build fails CI if any locale has > 20 missing keys.
- AC3: ICU plurals render correctly in Hindi (one/other) and Vietnamese (other).
- AC4: All 12 languages render without missing-glyph boxes on iOS, Android, and the web.
- AC5: A new untranslated key falls back to English silently in production but is reported by `i18n:scan` in CI.
